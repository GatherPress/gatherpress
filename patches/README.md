# Patches

This directory holds patches applied to `node_modules` by
[`patch-package`](https://www.npmjs.com/package/patch-package). Patches run
during `npm install` via the `postinstall` hook in `package.json`. Each patch
should be documented here so a future contributor knows what it does, why it
exists, and when it can be removed.

## `@wordpress+env+11.2.0.patch`

### What it changes

Adds two `composer global config` lines to the Docker image generated by
`@wordpress/env`'s `lib/runtime/docker/docker-config.js` — immediately before
the `composer global require --dev phpunit/phpunit` step:

```dockerfile
RUN composer global config --no-plugins audit.abandoned ignore
RUN composer global config --no-plugins audit.block-insecure false
```

### Why

Newer Composer versions run an audit on transitive dependencies during
`composer require`. PHPUnit's older supported tags (the `^5.7.21 || ^6.0 ||
^7.0 || ^8.0 || ^9.0 || ^10.0` range that wp-env installs) pull in deps
flagged by advisories `PKSA-5jz8-6tcw-pbk4` and `PKSA-z3gr-8qht-p93v`. The
audit aborts the install and the wp-env Docker build fails before the
container is ready, which means `npm run test:unit:php` cannot run.

The two `composer config` lines disable the audit's "abandoned" warning and
its "block on advisory" behavior so the install can proceed. Audit results
are not security-relevant inside the disposable test container — the
audit-bypass is local to the wp-env image build, not the plugin's own
Composer install.

### When to remove

Remove this patch — and the corresponding `patch-package` dependency +
`postinstall` hook in `package.json` if no other patches remain — once any of
the following is true:

- `@wordpress/env` upstream lands an option to disable Composer audit during
  the PHPUnit install step.
- `@wordpress/env` upstream pins a PHPUnit range that no longer triggers the
  advisories above.
- The `PKSA-5jz8-6tcw-pbk4` / `PKSA-z3gr-8qht-p93v` advisories are resolved
  upstream so the audit no longer blocks install.

After removing the patch file, bump `@wordpress/env` to a version that
includes the upstream fix and run `npm install` to regenerate the lockfile.

## Adding a new patch

1. Make the change inside `node_modules/<package>/...`.
2. Run `npx patch-package <package>` — it generates
   `patches/<package>+<version>.patch`.
3. Add a section to this README describing what the patch changes, why, and
   when it can be removed.
4. Commit the patch and the README update in the same change.
